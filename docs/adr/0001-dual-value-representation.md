# ADR-0001: Два раздельных типа Value (интерпретатор и VM)

Дата: 2026-07-23. Статус: принято.

## Контекст

В воркспейсе два исполняющих бэкенда с независимыми представлениями значений:

- `yps-interpreter`: `enum Value` в `crates/yps-interpreter/src/value.rs` — `String(String)`, окружения на `Rc<RefCell<EnvFrame>>` с `HashMap`-биндингами, mark-sweep GC поверх реестра фреймов.
- `yps-vm`: `enum Value` в `crates/yps-vm/src/value.rs` — `Str(Rc<str>)`, слот-индексированные локальные, Lua-подобные upvalue, собственный mark-sweep GC (с 1.7.0).

Мост (`crates/yps-vm/src/bridge.rs`) конвертирует значения глубоким копированием с identity-кэшем; интерпретаторные значения живут в VM как `Value::Host`.

Цена раздельности проявилась в Фазах 0–1: каждая фича пишется дважды (instance-методы строк/массивов, enforcement frozen/sealed, коэрция toString, GC с queue-roots), а расхождения между бэкендами — главный источник багов (вложенный delete, Display символов, деструктуризация строк, proxy-delete форвардинг).

## Решение

**Оставляем два раздельных типа Value. Не унифицируем.**

Обоснование:

1. **Проект учебный, и контраст — его суть.** «Tree-walking интерпретатор vs байткод-VM» — это заявленная ценность репозитория (README). Общий Value навязал бы обоим бэкендам компромиссное представление и стёр бы разницу подходов, ради которой проект существует.
2. **Оптимальные представления действительно разные.** VM выигрывает от `Rc<str>`, слотов и плоских upvalue; интерпретатору нужен `EnvFrame` с именованным поиском для REPL, hoisting и динамических конструкций. Унификация — размер XL с риском регрессий по всей conformance-базе ради экономии, которую даёт и дисциплина.
3. **Инвариант паритета уже механизирован.** Расхождения ловятся не глазами, а сьютом: 129+ vm_conformance-фикстур с побайтовым сравнением, плюс golden-база против Node. Каждая двойная реализация обязана сопровождаться паритетной фикстурой — это работающий процесс, подтверждённый Фазой 1.

## Последствия

- Двойная реализация фич остаётся осознанной ценой. Правило: фича не считается готовой без vm_conformance-фикстуры (или явного, задокументированного отказа с ошибкой на VM-пути).
- Перф-оптимизации Фазы 2 применяются к каждому бэкенду в его идиоме: `Rc<str>` и слоты — интерпретатору (у VM уже есть), inline-кэши и folding — VM.
- Семантическая логика, не зависящая от представления (коэрции, протоколы итераторов, регэксп-движки), по возможности выносится в разделяемые функции — это уже происходит (коэрция частично разделена, регэкспы переиспользуются мостом).
- Пересмотр решения оправдан, только если появится третий потребитель представления (например, JIT или транспайлер с общим рантаймом) — тогда выделение общего `yps-runtime`-крейта станет дешевле, чем тройная реализация.
