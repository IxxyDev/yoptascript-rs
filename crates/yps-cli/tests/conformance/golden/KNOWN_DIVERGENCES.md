# Известные расхождения с Node.js

Этот файл фиксирует намеренные и технические расхождения conformance-батареи
YoptaScript-rs с эталонным поведением Node.js. Раннер `tests/conformance.rs`
сверяет stdout кейсов из `cases/` с зафиксированным yopta-выводом в `golden/`,
а НЕ с живым Node.

Рабочий процесс для нового кейса:
1. Написать `cases/<name>.yop`.
2. Зафиксировать golden: `YPS_CONFORMANCE_BLESS=1 cargo test -p yps-cli`.
3. Написать `mirror/<name>.js` — Node-зеркало, печатающее тот же вывод через `console.log`.
   Если расхождение с Node намеренно, добавить первой строкой:
   `// DIVERGENCE: KNOWN_DIVERGENCES.md §<n> <краткая причина>`
   и внести запись в соответствующий раздел этого файла.
4. Запустить `node tools/gen-golden.js` — инструмент сравнивает вывод Node
   (из `mirror/`) с golden-файлами и сообщает: OK, MISMATCH, DOCUMENTED или SKIP.
   Завершается с ненулевым кодом при наличии недокументированных расхождений.
5. При расхождении: исправить баг в интерпретаторе и перезафиксировать golden,
   либо добавить нумерованную запись в этот файл и заголовок DIVERGENCE в mirror.

Node используется только офлайн-инструментом `tools/gen-golden.js` (вручную
разработчиком, не в CI).

## 1. Намеренные расхождения с Node (остаются по дизайну)

### 1.1. Оператор `из` (`in`) использует Display-ключ
`eval_binary` для `BinaryOp::In` (eval_expr.rs) берёт ключ через `left.to_string()`
(Display), а не через ECMA ToPropertyKey. Для объектных/массивных ключей это может
отличаться от Node. Изменение поведения вне рамок фазы 15.

### 1.2. Циклические импорты: пустой снимок вместо live-binding
При циклических `спиздить` модуль, ещё находящийся в состоянии `Loading`, виден из
импортёра как ПУСТОЙ снимок экспортов (фаза 15D, ModuleState{Loading,Loaded}): `Loading`
инициализируется пустой картой и не наполняется инкрементально. Node даёт «живые»
биндинги ES-модулей (импортёр в цикле видит дозаписи экспортёра); yopta возвращает
снимок на момент входа в цикл — то есть пустой. Главная цель 15D — устранить
бесконечную рекурсию/переполнение стека — достигнута; полный live-binding не реализован
осознанно (вариант D1). Это выбор рантайма, а не баг.

### 1.3. Слабые коллекции: Rc-семантика вместо настоящего GC
`СлабаяКарта`/`СлабыйНабор`/`СлабаяСсылка`/`РеестрФинализации` построены на
`std::rc::Weak`, а не на трассирующем сборщике мусора:
- Нет эфемеронной семантики: если значение в `СлабаяКарта` ссылается на свой
  собственный ключ (напрямую или через цикл), пара никогда не освободится —
  V8 такие пары собирает.
- Момент «смерти» детерминирован: объект умирает, когда падает последняя
  сильная `Rc`-ссылка (обычно при выходе из области видимости), а колбэки
  `РеестрФинализации` срабатывают на ближайшей итерации цикла событий —
  в Node момент вызова не специфицирован и может не наступить вовсе.
- Ключами/целями могут быть только объект, массив, `Карта` и `Набор`;
  функции и непрописанные символы, допустимые в V8, не поддерживаются.

### 1.4. RegExp: лимит бэктрекинга вместо зависания
Lookahead/lookbehind/backreferences исполняются движком `fancy-regex` (паттерны без них —
быстрым `regex`). У fancy-движка выставлен явный `backtrack_limit` (1 000 000):
катастрофически бэктрекающий паттерн в yopta завершается перехватываемым русским
`Косяк`-ом («Ошибка выполнения regex: …»), тогда как V8 может виснуть на таком паттерне
неограниченно долго. Это осознанный выбор рантайма (защита от враждебного ввода),
а не баг.

## 2. Технические расхождения (порядок/окружение) — НЕ покрываются golden

### 2.1. Порядок ключей объекта недетерминирован
`Value::Object` — `HashMap<String, Value>`, порядок итерации не стабилен между
запусками. Кейсы, печатающие многоключевые объекты через Display, `Объект.ключи`,
`Жсон.вСтроку`, `Отражение.собственныеКлючи` или объекты-результаты итераторов
(`{значение, готово}`), исключены из батареи. Затронутые примеры:
`stdlib`, `typed_arrays`, `regex`, `iterator_helpers`.

### 2.2. Зависимость от окружения/часов/путей
Не включены в golden как нестабильные между машинами и CI:
- `date` — методы локального времени и `сказать.время` (таймер);
- `io_demo` — `Процесс.рабочаяПапка`, `Процесс.перем`, `сказать.время`;
- `stack_trace` — встроенные в трассу абсолютные/относительные пути файла, вывод в stderr.

## 3. Коэрсивные golden приведены к Node (шаг 3 / 15C)

После шага 3 (abstract_equals + Add через ToPrimitive) golden-файлы
`coercion_equality`, `coercion_add`, `coercion_stringify` сверены с эталоном Node
(`tools/gen-golden.js`) и совпадают с ним полностью:

- `1 == "1"`, `ноль == неибу`, `правда == 1`, `0 == лож`, `"" == 0` → `true`;
- `"об=" + об` → `об=[object Object]`; `"мас=" + мас` → `мас=1,2,3`.

### 3.1. Встроенная `число` (Number()) — исправлена, совпадает с Node
Ранее `число(x)` возвращала `ноль` вместо `NaN` на неразбираемой строке, не делала
trim и давала `ноль` для `null`/`undefined`. Теперь `число` делегирует в
`coercion::to_number` (ECMA ToNumber): trim, `""`→`0`, `null`→`0`, `undefined`→`NaN`,
`0x`/`0o`/`0b`/`Infinity`, неразбираемое→`NaN`; `BigInt`→числовое значение. golden
`coercion_stringify` приведён к выводу Node. Расхождения нет.

## 4. Операторы, коэрсия, числа (ops_*/num_*)

### 4.1. Унарный минус связывается крепче `**`
`-2 ** 2` в yopta парсится как `(-2) ** 2` и даёт `4` — унарный минус привязывается
к основанию степени. В JS выражение `-2 ** 2` — синтаксическая ошибка (стандарт
запрещает нескобочный унарный оператор перед `**`, чтобы устранить неоднозначность);
по таблице приоритетов JS трактовал бы его как `-(2 ** 2) = -4`. Зеркало печатает
`-(2 ** 2)` (валидный JS), поэтому строки расходятся. Это выбор парсера yopta, а не баг.
Кейс `ops_exp_unary_minus`.

### 4.2. Реляционные операторы требуют двух чисел
`<`, `<=`, `>`, `>=` (`compare_op`) принимают только пару `Number`; для любых иных
операндов поднимается перехватываемая ошибка «Сравнение требует числа …». ECMA-коэрсия
(строковое сравнение `"10" < "9"`, числовое `"10" < 9`, `null`/`undefined`/`булево`
через ToNumber) НЕ выполняется. Это осознанная строго-числовая семантика (та же
философия, что у деления на ноль §4.4), а не баг. Кейс печатает `ошибка` там, где
Node даёт `true`/`false`; зеркало печатает реальные булевы значения Node → DOCUMENTED.
Кейс `ops_relational_strict`.

### 4.3. Унарные `+`/`-` требуют числа
`+x` и `-x` (`eval_unary`) определены только для `Number`/`BigInt`; для строки, булева,
`ноль`, `неибу` поднимается перехватываемая ошибка «Нельзя применить …». ECMA ToNumber
(`+"5"` → `5`, `+""` → `0`, `+"abc"` → `NaN`, `+правда` → `1`) НЕ выполняется. Та же
строго-числовая семантика. Кейс `ops_unary_numeric_strict`.

### 4.4. Деление на ноль поднимает ошибку
`a / 0` (для `Number`) даёт перехватываемую ошибку «Деление на ноль», а не
`Infinity`/`-Infinity`/`NaN`, как в Node. Осознанная защита от «тихих» бесконечностей
(`eval_expr.rs`, арм `BinaryOp::Div`). Остаток `a % 0` НЕ затронут и корректно даёт `NaN`.
Кейс `num_div_by_zero`.

### 4.5. Экспоненциальная нотация при печати чисел не поддерживается
Числовые литералы в лексере не понимают экспоненту/hex (`1e21`, `0x10` лексируются как
`1`+идентификатор); такие значения достижимы только через `число("1e21")`. При печати
`сказать`/`строка` yopta использует полную десятичную запись Rust: `число("1e21")` →
`1000000000000000000000` (Node: `1e+21`), `число("1e-7")` → `0.0000001` (Node: `1e-7`).
Порог перехода V8 на экспоненту (`≥ 1e21`, `≤ 1e-7`) не реализован осознанно — это
дополнительная фича форматирования, а не баг коэрсии. Целые в диапазоне `[2^53, 1e21)`
печатаются корректно полными цифрами (баг сатурации `as i64` исправлен, см. регрессии
`number_to_string_*`). Этот пункт НЕ покрыт golden-кейсом (нет способа создать такой
литерал в `.yop`).

## 5. Строки и RegExp (str_*/re_*)

### 5.1. Строки индексируются в скалярных значениях Unicode, а не в UTF-16
В V8 строка — это последовательность UTF-16 code unit'ов: `.length`, `at`, `charAt`,
`slice` и прочие индексные операции считают суррогатные пары как два элемента
(`"😀".length === 2`, `"a😀b".length === 4`). В yopta строка хранится как Rust `String`
(UTF-8) и все эти операции работают по `chars()` — то есть по скалярным значениям
Unicode: `"😀".длина === 1`, `"a😀b".длина === 3`. Для символов BMP (включая кириллицу
и `café`, где `é` — один code unit) поведение совпадает; расходится только на символах
вне BMP (эмодзи, редкие иероглифы). Доступ `"a😀b".at(1)` в Node даёт одиночный
суррогат `\ud83d`, в yopta — целый символ `😀`. Это осознанный выбор рантайма
(строки UTF-8, без суррогатной модели), а не баг. Кейс `str_case_length`.

## 7. Объекты, классы, деструктуризация (obj_*/cls_*)

Все 14 кейсов (`obj_getset`, `obj_getset_inherit`, `obj_computed_keys`, `obj_delete`,
`obj_spread_rest`, `obj_destructure`, `obj_optional_chain`, `obj_nullish`, `obj_freeze`,
`cls_inherit_super`, `cls_instanceof`, `cls_static`, `cls_this_binding`, `cls_three_level`)
сверены с Node. Расхождений нет, кроме §7.1 ниже.
В ходе написания батареи обнаружены и исправлены следующие баги интерпретатора:

- Статические методы/поля/геттеры/сеттеры НЕ наследовались подклассами: `Б.метод()`
  где `метод` объявлен `попонятия` в `А батя ...`, давало `undefined не является функцией`.
  Исправлено добавлением `find_static_method/field/getter/setter_in_class` с обходом
  родительской цепочки в `member.rs` и `assign.rs`.
- `яга.метод()` (super.method()) внутри экземплярного метода давало ошибку «super
  используется вне класса-наследника» — SUPER не связывался при вызове обычных методов.
  Исправлено через `call_method_with_this_super` с вычислением владельца метода и передачей
  родительского класса как super; обновлены вызовы методов на экземплярах, вызовы
  `super.метод()` и `яга()` (super-конструктор в цепочке 3+ уровней).
- Вызов `яга()` в цепочке 3 и более уровней (В→Б→А) уходил в бесконечную рекурсию —
  SUPER внутри тела конструктора Б был равен Б вместо А. Исправлено явной привязкой
  `grandparent` в обработчике `Expr::Super` как callee.
- `тырыпыры` (this) в `попонятия`-методе давало ошибку «this вне контекста объекта» —
  статические методы вызывались без привязки `this`. Исправлено: вызов статического
  метода через `call_method_with_this_super` с `this=Class`.
- Поле-стрелка (`стрела = () => тырыпыры.х`) не захватывало живой экземпляр: при
  извлечении и вызове возвращало `undefined`. Причина: `init_fields` передавал
  `Value::object(instance.clone())` — клон HashMap до оборачивания в `Rc`. Исправлено:
  `construct_instance` создаёт `Value::Object` до вызова `init_fields` и передаёт живой
  `Rc`-обёрнутый объект; стрелки закрываются над тем же Rc и видят все последующие записи.

### 7.1. `Кент.заморозить` (Object.freeze) не реализован
`Кент.заморозить` отсутствует в объекте `Кент`; обращение к нему даёт `undefined`,
вызов `undefined(...)` поднимает перехватываемую ошибку. В Node `Object.freeze(obj)`
блокирует запись свойств объекта: в строгом режиме — TypeError, в нестрогом — тихое
игнорирование. Реализация `freeze` требует добавления флага «заморожен» в хранилище
объекта (HashMap → структура с флагом) и проверки при записи — вне рамок данной
фазы. Кейс `obj_freeze` покрывает это расхождение (mirror DOCUMENTED).

### 7.2. Деструктурирующее присваивание к существующим переменным не поддерживается
Синтаксис `[x, y] = [y, x]` и `({ x } = obj)` (деструктуризация без объявления
переменной) не поддерживается парсером: LHS-паттерн не распознаётся как lvalue.
Деструктуризация в объявлениях (`гыы [а, б] = ...`) и в параметрах без паттерна
работает корректно. Не покрыто кейсом (функциональность парсера, не ошибка рантайма).

### 7.3. Деструктурирующие параметры с паттернами не поддерживаются
`йопта ф({ а = 1, б } = {}) { ... }` не парсится: `Param.name` — тип `Identifier`,
не `Pattern`. Не покрыто кейсом.

### 7.4. Мутация статических полей через `тырыпыры` или имя класса не поддерживается
`Счёт.число = Счёт.число + 1` в теле статического метода бросает ошибку «Нельзя
установить свойство у типа 'класс'»: `ClassDef.static_fields` — это `HashMap` внутри
`Rc<ClassDef>` без `RefCell`, поэтому мутация невозможна без реструктуризации
`ClassDef`. Статические поля доступны только для чтения после инициализации класса.
Не покрыто кейсом (батарея не тестирует мутацию статических полей).

## 6. Массивы и коллекции (arr_*/coll_*)

Все 13 кейсов (`arr_at`, `arr_concat_join`, `arr_flat`, `arr_immutable`, `arr_indexof`,
`arr_iteration`, `arr_reduce`, `arr_sort`, `arr_splice`, `coll_map`, `coll_map_nan_negzero`,
`coll_set`, `coll_setops`) сверены с Node и совпадают полностью. Расхождений нет.
В ходе написания батареи обнаружены и исправлены следующие баги интерпретатора:

- `indexOf` игнорировал отрицательный `fromIndex` (приводил к `usize`, теряя знак).
- `includes` полностью игнорировал второй аргумент `fromIndex`.
- `lastIndexOf` и `reduceRight` отсутствовали.
- `join` форматировал вложенные массивы через `Display` (`[2, 3]`) вместо рекурсивного
  join (ECMA `Array.prototype.join` рекурсивно вызывает `toString` → join на подмассивах).
- `Карта`/`Набор` хранили ключ `-0` в исходном виде; при итерации Node возвращает `0`
  (SameValueZero нормализует `-0` → `+0` при вставке). Исправлено через `MapKey::new`.
- Callback-методы (`map`, `filter`, `forEach`, `find`, `findIndex`, `findLast`,
  `findLastIndex`, `some`, `every`, `flatMap`, `reduce`, `reduceRight`) не передавали
  массив как третий аргумент колбэку — теперь передают `Value::Array(rc.clone())`.

Намеренных расхождений в этой батарее нет.

## 8. Итераторы, генераторы, промисы (gen_*/async_*)

15 кейсов (`gen_manual_iterator`, `gen_basics_two_way`, `gen_return_method`,
`gen_throw_method`, `gen_yield_delegate`, `gen_iterator_helpers_lazy`, `gen_for_of_break`,
`async_then_chaining`, `async_finally_semantics`, `async_microtask_order`,
`async_microtask_nested`, `async_all_allsettled`, `async_any_race`, `async_await_basics`,
`async_await_timer_order`) сверены с Node. Все совпадают с эталоном, кроме §8.2
(`async_await_timer_order`, mirror DOCUMENTED).

В ходе написания батареи обнаружены и исправлены следующие баги интерпретатора:

- **Двусторонняя связь `next(arg)` не работала.** `генератор.следующий(значение)`
  игнорировал аргумент: метод `следующий`/`next` в `stdlib/iterator.rs` жёстко
  передавал `GenInput::Send(Value::Undefined)` вместо первого аргумента. Выражение
  `гыы а = поебалу X` всегда давало `а = undefined`. Исправлено: берётся
  `args.into_iter().next()`. Первый `next()` по-прежнему отбрасывает аргумент (ECMA).
  Регрессия: `generator_next_arg_two_way_communication`.

- **`return`/`throw` в `тюряжка` зависали при `генератор.вернуть()`.** Когда генератор
  приостановлен в `try`, а `finally` сам делает `отвечаю`/`кидай`, повторный вход в
  `finally` в `unwind` (арм `GenFrame::TryCatch`, ветви Return/Break/Continue)
  бесконечно переустанавливал `finally`-блок. Исправлено проверкой `pending`
  (`state ∈ {Trying, InCatch}`) перед повторным запуском `finally`: если `finally`
  уже исполнялся, кадр `try` снимается и абортивное завершение распространяется наружу
  (новый `return`/`throw` переопределяет ожидающий). Регрессии:
  `generator_return_in_finally_overrides_gen_return`,
  `generator_throw_in_finally_overrides_gen_return`.

- **`поебалуна` (yield\*) не прокидывал протокол в генератор-делегат.** Делегирование
  гналось через обобщённый `iterator::next`, который всегда шлёт `Undefined` и не умеет
  `return`/`throw`. Следствия: (а) посланные значения не доходили до внутреннего
  генератора; (б) `генератор.вернуть()` сквозь `yield*` не запускал `finally` внутреннего
  генератора; (в) `генератор.кинуть()` сквозь `yield*` не давал внутреннему `try/catch`
  перехватить и продолжить. Исправлено новым `delegate_step` (драйвит внутренний
  `IteratorState::Generator` через `step_generator` с нужным `GenInput`) и полем
  `GenState.pending_send` для проброса значения `Send`; арм `GenFrame::Delegate` в
  `pump` и `unwind` теперь прокидывает next/return/throw. Регрессии:
  `generator_yield_delegate_forwards_sent_values`,
  `generator_yield_delegate_return_runs_inner_finally`,
  `generator_yield_delegate_throw_recovers_in_inner_catch`.

- **`го`-цикл не закрывал генератор при раннем `харэ`.** `го (… сашаГрей ген())` с
  `харэ`/`return`/throw из тела не вызывал `генератор.вернуть()`, поэтому `finally`
  внутри генератора не исполнялся (ECMA закрывает итератор при досрочном выходе из
  for-of). Добавлен `iterator::close` (драйвит генератор `GenInput::Return`, рекурсивно
  заходит во внутренние итераторы адаптеров `map`/`filter`/`take`/`drop`); вызывается в
  `exec_for_of_loop` на ветвях Break/Exit и при ошибке тела. Регрессия:
  `generator_closed_when_for_of_breaks_early`.

### 8.1. Микротаски на верхнем уровне сливаются после каждого оператора
`Interpreter::run` (mod.rs) после каждого оператора верхнего уровня вызывает
`drain_microtasks`, тогда как в Node весь синхронный скрипт исполняется целиком,
и только потом стартует очередь микротасок. Поэтому на верхнем уровне
`СловоПацана.решить(1).потом(cb); сказать("после")` напечатает результат `потом`
ДО `после`, а классический паттерн `1 4 2 3` ломается на `1 2 3 4`. Это осознанная
модель рантайма (REPL опирается на пооператорный слив, чтобы печатать промежуточные
результаты), а не баг. Обход в conformance-кейсах: вся промис-логика обёрнута в
функцию/`ассо`-функцию (тело выполняется синхронно одним top-level оператором, слив
происходит после него) — тогда микротаск-порядок совпадает с Node, что и подтверждают
`async_microtask_order`, `async_microtask_nested`, `async_then_chaining`. Отдельным
golden-кейсом верхнеуровневое расхождение не покрывается (вместо этого продемонстрирован
корректный путь через обёртку).

### 8.2. `сидетьНахуй` (await) синхронен — не возвращает управление вызывающему
`сидетьНахуй` блокирует выполнение текущей функции до разрешения обещания (сливая
очереди микро-/макротасок на месте), а не приостанавливает `ассо`-функцию как
продолжение, отдавая управление вызывающему. Следствия в `async_await_timer_order`:
(1) код после `демо()` (`сказать("после main")`) исполняется ПОСЛЕ полного завершения
`main`, тогда как в Node `main` отдаёт управление на первом `await`, и `после main`
печатается сразу за `старт`; (2) код после `await` внутри `ассо`-функции исполняется
синхронно при разрешении, а не как отложенная микротаска. Порядок слива микро-/макротасок
ВНУТРИ функции при этом совпадает с Node (микротаски до таймеров, таймеры по задержке).
Это фундаментальный выбор рантайма (на нём же построены top-level await и `сумма: 6` в
`example_event_loop`), а не баг. Кейс `async_await_timer_order` (mirror DOCUMENTED).
Все прочие async-кейсы тестируют РЕЗУЛЬТАТ await (значения, перехват reject через
`try/catch` и `ловить`, формы `всех`/`всехУстаканить`/`любой`/`гонка`), который от этого
расхождения не зависит и совпадает с Node.

Прочие замечания (не баги, известные ограничения, кейсами не покрыты):
- `поебалуна` (yield\*) запрещён в декларации/присваивании (`гыы x = поебалуна g()`):
  парсер требует отдельный оператор. Поэтому возвращаемое значение `yield*`-выражения
  (значение `return` внутреннего генератора) в yopta захватить нельзя; значения проброса
  и закрытие делегата при этом работают (см. `gen_yield_delegate`).
- `СловоПацана.любой` при полном отказе бросает объект-агрегат с русским `message`
  («Все обещания отклонены») и полем `errors`; это не настоящий `AggregateError`-инстанс
  и `тип(e)` даёт `объект`. Кейс печатает только `errors.длина` и элементы поэлементно
  (совпадает с Node), но не `message`/`тип`.
